# Звіт до лабораторної роботи №5: Контекст та Авторизація

## Фрагменти коду основних механізмів

### AuthContext.jsx — Глобальний стан авторизації
```jsx
export const AuthProvider = ({ children }) => {
  const [isAuthenticated, setIsAuthenticated] = useState(false);
  const [user, setUser] = useState(null);

  const login = (email) => {
    setIsAuthenticated(true);
    setUser({ email });
  };

  const logout = () => {
    setIsAuthenticated(false);
    setUser(null);
  };

  return (
    <AuthContext.Provider value={{ isAuthenticated, user, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
};
```

### ProtectedRoute.jsx — HOC для захисту маршрутів
```jsx
const ProtectedRoute = ({ children }) => {
  const { isAuthenticated } = useAuth();

  if (!isAuthenticated) {
    return <Navigate to="/login" replace={true} />;
  }

  return children ? children : <Outlet />;
};
```

## Відповіді на контрольні запитання

**1. Яку архітектурну проблему вирішує Context API?**
Context API вирішує проблему «prop drilling» — ситуацію, коли дані (наприклад, стан авторизації або тема) потрібно передати через багато рівнів вкладеності компонентів, навіть якщо проміжним компонентам ці дані не потрібні. Контекст дозволяє зробити дані доступними будь-якому компоненту в дереві без необхідності передавати їх через пропси на кожному рівні.

**2. Чому для великих проектів часто обирають Redux або Zustand замість Context API?**
Context API не оптимізований для частих оновлень великої кількості незалежних даних: при кожній зміні значення контексту перерендериться все дерево компонентів, які підписані на нього. Redux та Zustand мають кращі механізми вибору (selectors), що дозволяє оновлювати лише конкретні підписані вузли, а також пропонують потужні DevTools та чітку структуру для управління складним станом.

**3. У чому полягає роль HOC при реалізації захищених маршрутів?**
Роль Higher-Order Component (HOC) — огортати цільові компоненти та реалізовувати логіку перевірки умов доступу (наприклад, `isAuthenticated`) перед їх відображенням. Це дозволяє централізовано керувати безпекою маршрутів, автоматично перенаправляючи неавторизованих користувачів на сторінку логіну та запобігаючи витоку приватної інформації.

**4. Навіщо використовувати параметр replace: true при перенаправленні після логіну?**
Параметр `replace: true` замінює поточний запис у стеку історії браузера новим. Це важливо після успішної авторизації, щоб сторінка логіну «зникла» з історії переходів. Це не дозволяє користувачеві повернутися на екран логіну простим натисканням кнопки «Назад» у браузері, що забезпечує коректну та безпечну логіку навігації.
