# Звіт до лабораторної роботи №3

## Код логіки пошуку та фільтрації за категорією
```jsx
const [searchTerm, setSearchTerm] = useState('');
const [activeCategory, setActiveCategory] = useState('ALL');

const filteredPosts = postsData.filter((post) => {
  const matchesSearch =
    post.content.toLowerCase().includes(searchTerm.toLowerCase()) ||
    post.author.toLowerCase().includes(searchTerm.toLowerCase());
  const matchesCategory =
    activeCategory === 'ALL' || post.category === activeCategory;
  return matchesSearch && matchesCategory;
});
```

## Обробка порожнього результату
Ми використовуємо умовний рендеринг за допомогою тернарного оператора. Якщо масив `filteredPosts` має довжину більше 0, ми ітеруємо по ньому за допомогою `.map()`. Якщо ж довжина масиву дорівнює 0 (результатів не знайдено), ми виводимо стилізований блок `div` із повідомленням про системну помилку та червоним неоновим світінням.

```jsx
<div style={{ width: '100%', maxWidth: '600px' }}>
  {filteredPosts.length > 0 ? (
    filteredPosts.map((post) => <Post key={post.id} {...post} />)
  ) : (
    <div style={{ /* стилі помилки */ }}>
      Системна помилка: Записів за вашим запитом не знайдено.
    </div>
  )}
</div>
```

## Відповіді на контрольні запитання

**1. Що таке "підняття стану" і чому воно використовується для SearchBar?**
"Підняття стану" (Lifting State Up) — це патерн у React, коли стан передається від дочірнього компонента до спільного батьківського компонента. Для `SearchBar` ми піднімаємо стан `searchTerm` в `App.jsx`, тому що саме в `App` знаходиться масив даних, який потрібно фільтрувати. Це дозволяє іншим компонентам (списку постів) миттєво реагувати на зміни, що відбуваються в полі пошуку.

**2. Пояснення асинхронної природи оновлення стану в useState.**
React групує оновлення стану у пакети (batching) для оптимізації продуктивності, щоб уникнути зайвих перемальовувань (re-renders). Через це нове значення стану не стає доступним миттєво в наступному рядку коду після виклику функції-сеттера. Замість цього React планує оновлення і виконає його лише під час наступного циклу рендерингу.

**3. Чому для фільтрації створюється нова змінна filteredPosts замість зміни оригінального масиву?**
Ми створюємо нову змінну, щоб уникнути мутації (зміни) початкових даних. Якщо ми змінили б оригінальний масив `postsData`, ми б назавжди втратили ті елементи, які не пройшли фільтр, і не змогли б "скасувати" пошук або перемкнути категорію назад. Зберігаючи оригінал незмінним, ми можемо динамічно відображати будь-який зріз даних на основі поточних стейтів.

**4. У чому перевага використання керованих компонентів над некерованими при реалізації пошуку?**
Керовані компоненти використовують стан React як єдине джерело істини. Це дає змогу миттєво перехоплювати кожне натискання клавіші (через `onChange`) та одразу ж оновлювати стан і фільтрувати список. Нам не потрібно вручну звертатися до DOM-елемента через `ref` або чекати надіслання форми, що забезпечує плавний та "живий" пошук для користувача.
