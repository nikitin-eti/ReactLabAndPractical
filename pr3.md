# Звіт до практичної роботи №3

## Код логіки умовного рендерингу

### Використання оператора `&&` для довідки
```jsx
{showHelp && (
  <div style={{ /* стилі голографічної панелі */ }}>
    <p>Довідка: Дозволяє керувати списками студентів.</p>
  </div>
)}
```

### Використання тернарного оператора для вибору списку та статусу
```jsx
const displayedStudents = filterActive ? students.filter(s => s.score >= 60) : students;

// У JSX:
<span style={{ color: (student.score ?? 0) >= 60 ? '#10b981' : '#ef4444' }}>
  [{ (student.score ?? -1) === -1 ? 'Дані пошкоджено' : (student.score >= 60 ? 'Зараховано' : 'Незараховано') }]
</span>
```

### Система табів (умовний рендеринг компонентів)
```jsx
{activeTab === 'list' && <StudentList students={students} filterActive={filterActive} />}
{activeTab === 'stats' && <StatisticsData students={students} />}
{activeTab === 'author' && <AboutAuthor />}
```

## Відповіді на контрольні запитання

**1. Чому використання if/else безпосередньо всередині JSX-розмітки призведе до помилки?**
JSX — це синтаксичний цукор над викликами функцій `React.createElement()`. В JavaScript вирази (expressions) можуть бути аргументами функцій, а інструкції (statements), такі як `if/else`, — ні. Всередині фігурних дужок у JSX можна використовувати лише вирази, які повертають значення.

**2. Яка небезпека криється у використанні числа 0 у лівій частині оператора && в JSX?**
В JavaScript `0` є хибним (falsy) значенням, але в JSX, якщо ліва частина оператора `&&` дорівнює `0`, React відрендерить це число `0` на екрані. Це часто стає причиною появи небажаних нулів у інтерфейсі замість очікуваного порожнього місця. Для уникнення цього краще примусово перетворювати значення на булеве: `!!array.length && ...`.

**3. Як реалізувати умовний рендеринг при складній логіці (5+ варіантів)?**
Для складної логіки краще використовувати конструкцію `switch` або об'єкт-мапу (словник) компонентів поза JSX-розміткою. Це дозволяє зробити код чистішим та легшим для читання, оскільки велика кількість вкладених тернарних операторів стає некерованою.

**4. В яких випадках використання тернарного оператора є кращим за &&?**
Тернарний оператор є кращим, коли потрібно відрендерити два взаємовиключні блоки UI («або елемент А, або елемент Б»). Оператор `&&` підходить лише для випадків «або елемент, або нічого» (short-circuit evaluation).

**5. Як React розуміє, що потрібно перерендерити частину інтерфейсу при зміні вкладки?**
При зміні стану через функцію-сеттер (наприклад, `setActiveTab`), React отримує сигнал про необхідність оновлення. Він запускає процес узгодження (Reconciliation), порівнює попереднє віртуальне дерево DOM із новим, виявляє різницю і точково оновлює лише ті частини реального DOM-дерева, які змінилися відповідно до нової логіки таба.
