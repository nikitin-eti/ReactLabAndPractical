# Звіт до лабораторної роботи №4: Маршрутизація

## Структура маршрутів

У додатку реалізовано наступну архітектуру маршрутизації:
- `/` — Головний макет (`MainLayout`) з навігаційною панеллю.
  - (index) — Вітальна сторінка `Home`.
  - `feed` — Стрічка постів-логів `Feed`.
  - `feed/:postId` — Детальна сторінка запису `PostPage`.
  - `profile/*` — Профіль оперативника з вкладеними маршрутами.
- `*` — Сторінка 404 `NotFound` для неіснуючих секторів.

## Фрагменти коду

### Конфігурація маршрутів у `App.jsx`
```jsx
<Routes>
  <Route path="/" element={<MainLayout />}>
    <Route index element={<Home />} />
    <Route path="feed" element={<Feed />} />
    <Route path="feed/:postId" element={<PostPage />} />
    <Route path="profile/*" element={<Profile />} />
    <Route path="*" element={<NotFound />} />
  </Route>
</Routes>
```

### Використання `Outlet` та `NavLink` у `MainLayout`
```jsx
<nav className={styles.nav}>
  <NavLink to="/feed" className={({ isActive }) => isActive ? styles.active : styles.link}>
    ЛОГИ_МЕРЕЖІ
  </NavLink>
</nav>
<main>
  <Outlet />
</main>
```

### Логіка `useParams` та `useNavigate` у `PostPage`
```jsx
const { postId } = useParams();
const navigate = useNavigate();
const post = postsData.find((p) => p.id === Number(postId));

// Повернення назад:
<Button onClick={() => navigate(-1)}>ПЕРЕРВАТИ_З'ЄДНАННЯ</Button>
```

### Програмна навігація у `ProfileSettings`
```jsx
const navigate = useNavigate();
// ...
<Button onClick={() => navigate('/')}>ЗБЕРЕГТИ КОНФІГУРАЦІЮ</Button>
```

## Відповіді на контрольні запитання

**1. У чому різниця між клієнтською та серверною маршрутизацією?**
Клієнтська маршрутизація маніпулює DOM-деревом локально в браузері без запиту всього нового документа із сервера. Сторінка не перезавантажується повністю, що забезпечує миттєвий перехід між розділами. Серверна маршрутизація вимагає повного перезавантаження сторінки та отримання нового HTML-файлу від сервера при кожному переході.

**2. Навіщо потрібен атрибут index у маршруті?**
Атрибут `index` вказує, що даний дочірній маршрут є типовим (стартовим) для батьківського маршруту. Він відображається в батьківському компоненті `Outlet` за замовчуванням, коли URL збігається з шляхом батька.

**3. Які переваги використання хука useNavigate порівняно з компонентом Link?**
Хук `useNavigate` повертає функцію, яку можна викликати всередині логіки (наприклад, після успішної валідації форми, завершення API-запиту або виконання певних обчислень). Це забезпечує гнучкість управління потоком програми («програмна навігація»), на відміну від компонента `Link`, який призначений лише для прямої візуальної навігації за кліком користувача.

**4. Як реалізувати динамічне підсвічування активного посилання в меню?**
Динамічне підсвічування реалізується через компонент `NavLink`. Його атрибути `className` або `style` можуть приймати функцію, яка отримує об'єкт зі станом `isActive`. Залежно від цього стану можна повертати різні рядки класів або об'єкти стилів для візуального виділення активного пункту.

**5. Що робить маршрут зі шляхом *?**
Маршрут зі шляхом `*` діє як «пастка» (catch-all). Він активується, коли жоден інший маршрут у списку не збігається з поточним URL. Зазвичай він використовується для відображення сторінки `NotFound` (404) і обов'язково розміщується останнім у списку `Routes`.
